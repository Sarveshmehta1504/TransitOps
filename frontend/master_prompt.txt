## 1. UNDERSTAND THE BACKEND FIRST (read-only — do not edit anything here)

Before writing any frontend code, spend real time actually reading the backend so the frontend isn't guessing at shapes. Specifically:

- Open `backend/routes/api.php` (or wherever API routes are registered) and list every existing route: method, path, controller, and whether it's behind auth middleware.
- Open the corresponding controllers in `backend/app/Http/Controllers/` for each route and note the actual request validation rules and the actual JSON response shape (field names, nesting, pagination format) — don't assume Laravel defaults, read the real code.
- Open `backend/app/Models/` for Vehicle, Driver, Trip, Maintenance, FuelLog/Expense, User (and Role, if it's a separate model or just a column) — note real column names, casts, relationships (e.g. does Trip belong to Vehicle and Driver, does Vehicle have many MaintenanceRecords).
- Check `backend/database/migrations/` if a model's fields are unclear from the model file alone — migrations are the ground truth for schema.
- Check how auth is implemented: Sanctum (SPA cookie-based) vs. Sanctum (token-based) vs. Passport vs. custom. Check `backend/config/sanctum.php` and the login/logout controller to see exactly what it expects and returns.
- Check `backend/config/cors.php` — note whether `http://localhost:3000` is already allowed. If not, don't edit it — tell me so it can be added on the backend side.
- Produce a short internal note (a comment block at the top of `frontend/src/lib/api/client.ts` is fine) summarizing: auth mechanism used, token storage expectation, and a table of real endpoints you found (method, path, purpose) — this becomes the contract the rest of the frontend is built against. Do not invent endpoints that don't exist; if something in the spec (e.g. CSV export) has no backend route yet, flag it explicitly instead of building against a guessed URL.

## 4. DESIGN APPROACH

- Use whatever design/prototyping tools and skills you have available (e.g. a design-system skill, a Stitch-style or frontend-design skill, or a "design-task-frontend" workflow if configured in this environment) to establish the visual direction *before* writing component code: define the color tokens, type scale, spacing scale, and component states (default/hover/active/disabled/loading/error/empty) once, in one place, and reuse them everywhere. ## 6. AUTH & RBAC

Roles: Fleet Manager, Driver, Safety Officer, Financial Analyst.

- Wire login/logout/session using exactly the mechanism found in Step 1 — do not guess.
- `middleware.ts` redirects unauthenticated users to `/login`.
- Route-group access: `settings/users` → Fleet Manager only. `reports` → Financial Analyst + Fleet Manager. `trips/new` → Driver + Fleet Manager. Safety Officer gets full visibility into drivers/compliance, read access elsewhere.
- Hide nav items the current role can't access AND guard the route itself (both layers).

## 7. DASHBOARD

KPI cards: Active Vehicles, Available Vehicles, Vehicles in Maintenance, Active Trips, Pending Trips, Drivers On Duty, Fleet Utilization (%).
Filter bar: vehicle type, status, region — client component, re-fetches via TanStack Query.
Two Recharts visuals: utilization trend + status breakdown, styled to the custom token set, not default Recharts look.

## 8. MODULES — BUILD IN THIS ORDER

1. **Auth** — login page + session wiring.
2. **Vehicle Registry** — TanStack Table: Reg No (unique), Name/Model, Type, Max Load, Odometer, Acquisition Cost, Status (Available/On Trip/In Shop/Retired, color-coded). Create/edit modal, Zod-validated. Detail page: linked trips, maintenance, fuel logs.
3. **Driver Management** — Name, License No, License Category, License Expiry (flag <30 days and expired distinctly), Contact, Safety Score, Status (Available/On Trip/Off Duty/Suspended).
4. **Trip Management** — lifecycle board (Draft → Dispatched → Completed → Cancelled). New Trip wizard: source, destination, only *available* vehicles, only *available, non-suspended, non-expired-license* drivers, cargo weight (client-side check against selected vehicle's max load, backend remains final authority), planned distance. Dispatch/Complete/Cancel call real backend endpoints; reflect status via TanStack Query mutations + refetch.
5. **Maintenance** — create maintenance record for a vehicle; render backend-driven status flip to In Shop, don't compute it client-side.
6. **Fuel & Expense Management** — log fuel (liters, cost, date) and expenses (tolls etc.); show total operational cost per vehicle exactly as the API returns it.
7. **Reports & Analytics** — Fuel Efficiency (Distance/Fuel), Fleet Utilization, Operational Cost, Vehicle ROI `(Revenue − (Maintenance + Fuel)) / Acquisition Cost`. CSV export client-side from fetched data. PDF export: disabled button with "coming soon" tooltip (bonus, skip for now).

## 9. API INTEGRATION

- Base URL from `NEXT_PUBLIC_API_URL` only — never hardcode a host in a component.
- One thin `lib/api/<entity>.ts` per entity, one matching `hooks/use-<entity>.ts` — components call hooks only, never `lib/api` directly.
- Shared error-parsing utility for Laravel's typical `{ message, errors: { field: [...] } }` shape so every form surfaces field errors the same way.
